#### **Introdução:**

Com base nas sugestões disponíveis no post [Linux kernel patch suggestions (FSD 2026)][suggestions], eu e meu colega de equipe, [Lucas Ciziks][lucas], optamos por contribuir com melhorias no uso de mutexes.

Mutexes são ferramentas utilizadas pelo kernel para impedir que múltiplas threads acessem o mesmo recurso ao mesmo tempo, com isso, eles ajudam a prevenir race conditions, deadlocks e outros tipos de inconsistências.

A maneira em que mutexes eram implementados no kernel, `mutex_lock(&lock)`/`mutex_unlock(&lock)`, abria margem para erros. Afinal era obrigatória a chamada da função `mutex_unlock(&lock)` para liberar o mutex, criando assim, a necessidade de vários unlocks em fluxos complexos ou até o uso da diretiva goto. Exemplo:

```c
{
    struct mutex lock;   // declaring a mutex
    mutex_init(&lock);   // dynamically initialising the mutex
    mutex_lock(&lock);   // acquiring the mutex
    ...                  // critical section
    mutex_unlock(&lock); // unlocking the mutex
}
```
###### Código obtido em: [Suggestion 1.1) Replace mutex_lock(&lock) and mutex_unlock(&lock) calls with guard(mutex)(&lock)][chapter1]

Mas tudo isso pode ser resolvido utilizando uma macro criada em 2023, o que é relativamente novo para o kernel, chamada `guard(mutex)(&lock)` que basicamente cria uma trava dentro do escopo em que ela está inserida, Exemplo:

```c
{
    struct mutex lock;   // declaring a mutex
    guard(mutex)(&lock); // acquiring the mutex
    ...                  // critical section
}                        // from here on, the mutex is unlocked
```

Também existe a possibilidade de usarmos uma macro similar chamada scoped_guard(mutex, &lock):

```c
{
    ...                          // do some stuff with no mutex active
    scoped_guard(mutex, &lock) { // lock is a struct mutex that has been initialised
    ...                          // inside these curly braces, the mutex is active
    }                            // from here on, the mutex is unlocked
    ...                          // do some other stuff with no mutex active
}
```
###### Código obtido em: [Suggestion 1.1) Replace mutex_lock(&lock) and mutex_unlock(&lock) calls with guard(mutex)(&lock)][chapter1]

Dado essa breve explicação, vamos ao patch...

-   mutex_lock(&adc->lock);

#### **v2:**

Ao longo dos dias 16/04/2026 e 17/04/2026 recebi alguns retornos com orientações e sujestões de melhoria para o código, sendo:

1. Se referir as funções como func() na mensagem de commit
2. Manter a ordem dos #includes
3. Elaborar melhor a alteração na função ingenic_adc_read_chan_info_raw()

A primeira parte foi fácil, ajustei a mensagem do commit e ordenei os #includes. Sobre a função ingenic_adc_read_chan_info_raw() resolvi substituir a macro guard() pela scoped_guard() para manter o comportamento original da função.
